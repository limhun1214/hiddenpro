# HiddenPro — 비즈니스 룰 및 어뷰징 방지 정책

> **문서 위치**: `docs/abuse_prevention.md`
> **적용 대상**: 안티그래비티(실행 에이전트) 및 아키텍츠(설계 에이전트) 모두
> **최종 업데이트**: 2026-03-22
> **목적**: 전문가-고객 매칭 안정성 보장 및 플랫폼 어뷰징 방지 핵심 로직 기록

## 1. 플랫폼 매칭 안정성 기본 원칙

### 1.1 과금 모델

- **견적 발송 즉시 Credits 차감** (에스크로/후정산 방식 금지)
- Credits = 플랫폼 내부 가상 토큰. UI 표기: `Top-up Credits`, `Credit History`, `Available Credits`
- 법정 화폐(₱ PHP) 표기 코드/UI에 직접 사용 금지

### 1.2 핵심 설계 원칙

```
신뢰도 = (매칭 완료율) × (양방향 어뷰징 방지) × (투명한 정산)
```

견적 요청 → 견적 발송 → 매칭 확정 → 서비스 완료 → 리뷰 의 전체 파이프라인에서 어느 단계도 임의 우회 불가.

## 2. 고객 견적 관리 로직 (Customer Quote Management)

### 2.1 견적 요청 상태 머신

```
OPEN (견적 대기)
  │
  ├─ 견적 수신 (1~5개) ─────────────────→ OPEN (견적 있음)
  │
  ├─ 최대 견적 수 도달 (현재 5개) ──────→ OPEN (잠김 — 신규 견적 발송 불가)
  │
  ├─ 고객이 1개 낙찰 ───────────────────→ MATCHED
  │
  ├─ 48시간 미결정 (무응답 자동 만료) ──→ EXPIRED → 마감된 견적
  │
  └─ 고객이 직접 취소 ──────────────────→ CANCELED → 마감된 견적
```

### 2.2 탭 분류 기준 (고객 화면 — match_requests 기준)

| 상태 / 조건                                          | 올바른 탭      |
| ---------------------------------------------------- | -------------- |
| OPEN (견적 0개 / 대기 중)                            | 진행 중인 견적 |
| OPEN + 견적 있음                                     | 진행 중인 견적 |
| 견적 최대 개수 도달 (5개)                            | 진행 중인 견적 |
| MATCHED (확정 완료)                                  | 진행 중인 견적 |
| 리뷰 작성 완료                                       | 마감된 견적    |
| MATCHED 확정 후 30일 경과 (리뷰 미작성 시 자동 만료) | 마감된 견적    |
| CANCELED                                             | 마감된 견적    |
| 48시간 만료                                          | 마감된 견적    |

### 2.3 고수(Pro) 화면 탭 분류 (match_quotes 기준)

| 상태 / 조건                            | 탭 위치            | 동작 기준                              |
| -------------------------------------- | ------------------ | -------------------------------------- |
| 요청서 발생 (내 견적 미발송)           | 새로운 요청        | 견적 발송 즉시 이 탭에서 제거됨        |
| 견적 발송 완료 (대기 중)               | 보낸 견적 > 진행중 | 발송 즉시 진입                         |
| 내가 낙찰됨 (ACCEPTED)                 | 보낸 견적 > 진행중 | 매칭 성공 후 유지                      |
| 고객 리뷰 작성 완료                    | 보낸 견적 > 보관함 | 서비스 완료 간주                       |
| 낙찰 후 30일 경과                      | 보낸 견적 > 보관함 | 리뷰 미작성 무한 적재 방지             |
| 다른 고수 낙찰 (나는 탈락 — IACCEPTED) | 보낸 견적 > 보관함 | 즉시 이동                              |
| 고객 무응답 48시간 만료                | 보낸 견적 > 보관함 | 자동 이동                              |
| 고객이 취소 (CANCELED)                 | 보낸 견적 > 보관함 | 낙찰 전/후 관계없이 이동               |
| 보관함 이동 후 7일 경과                | 숨김 처리          | DB 쿼리 레벨 필터링 + 프론트 이중 방어 |

## 3. 어뷰징 방지 핵심 규칙 — 3-15-0 규칙

### 3.1 정의

**3-15-0 규칙**은 고수(Pro)가 Credits 소진 후에도 무제한 견적을 발송하거나, 자동 환경에서 스팸성 견적을 반복 발송하는 것을 방지하기 위한 3단계 방어 체계다.

| 번호   | 규칙 이름           | 내용                                                                                         |
| ------ | ------------------- | -------------------------------------------------------------------------------------------- |
| **3**  | 최대 견적 수 제한   | 하나의 요청서에 발송 가능한 고수의 최대 견적 수 = **5개** (설정값, 관리자 조정 가능)         |
| **15** | Credits 보유 임계값 | 고수가 견적 발송 전 보유해야 하는 최소 Credits 임계치 = default 추후 정의                    |
| **0**  | 환불 불가 원칙      | 고객이 낙찰하지 않거나 48시간 만료 시 **Credits 자동 환불** (단, 낙찰 후 취소는 정책별 상이) |

> **주의**: 3-15-0의 숫자는 현재 구현 기준이며, 관리자 패널의 `platform_settings` 테이블에서 런타임 조정 가능하다. 코드에 하드코딩 금지 — 반드시 DB 설정값을 참조할 것.

### 3.2 3-15-0 규칙 동작 원리

#### A. 최대 견적 수 제한 (3 — 실제 설정값: 5)

```sql
-- match_requests 테이블에서 해당 요청서의 견적 수를 확인
SELECT COUNT(*) FROM match_quotes
WHERE request_id = $request_id
  AND status != 'CANCELED';

-- 5개 이상이면 신규 견적 발송 RPC 차단
-- send_quote_and_deduct_cash() 내부에서 COUNT 체크 후 EXCEPTION 발생
```

**무결성 보장**: 프론트엔드 버튼 숨김만으로 제어하는 것이 아니라, **RPC 함수 레벨에서 DB가 직접 차단**한다. RLS 및 CHECK 제약 적용.

#### B. Credits 보유 임계값 검사 (15)

```sql
-- send_quote_and_deduct_cash() RPC 함수 내 구현
-- 트랜잭션 시작 → Credits 잔액 확인 → 부족 시 EXCEPTION → 롤백
BEGIN;
  SELECT available_credits FROM users WHERE id = pro_id FOR UPDATE;
  IF available_credits < quote_cost THEN
    RAISE EXCEPTION 'INSUFFICIENT_CREDITS';
  END IF;
  UPDATE users SET available_credits = available_credits - quote_cost WHERE id = pro_id;
  INSERT INTO cash_ledger (user_id, tx_type, amount, ...) VALUES (...);
COMMIT;
```

**동시성 방어**: `FOR UPDATE` 행 잠금으로 Race Condition(동시 차감) 방지.

#### C. 환불 원칙 (0 — 미낙찰 시 자동 환불)

- **환불 트리거**: 요청서 48시간 만료 또는 고객이 다른 고수를 낙찰
- **환불 RPC**: `refund_unread_quotes()` — 대상 요청서의 미낙찰 견적들에 Credits 복원
- **환불 불가 케이스**: 낙찰(ACCEPTED) 후 고수가 스스로 취소한 경우 (정책 별도 정의)
- **원장 기록**: 모든 환불은 `cash_ledger` 테이블에 `tx_type = 'REFUND'`로 기록

### 3.3 추가 어뷰징 방어 레이어

#### 추천 프로그램 어뷰징 방지

| 방어 항목            | 구현 방법                                                            |
| -------------------- | -------------------------------------------------------------------- |
| 자기 추천 불가       | DB: `referred_by != referrer_id` CHECK 제약                          |
| 중복 보상 불가       | DB: `referral_rewards` 테이블 UNIQUE (referrer_id, referred_user_id) |
| 가입만으로 보상 불가 | 트리거 조건: 피추천인의 첫 견적 발송/요청 완료 후에만 지급           |
| 계정 중복 생성 방지  | 전화번호 인증 필수 (가입 완료 전 SMS OTP 검증)                       |

#### 금융 로직 무결성

- `send_quote_and_deduct_cash()`: 견적 발송 + Credits 차감 원자적 처리
- `refund_unread_quotes()`: 미낙찰 견적 Credits 자동 환불
- `admin_manage_cash()`: 관리자 Credits 수동 조정 (감사 로그 포함)
- `admin_process_payout()`: 고수 정산 처리

**철칙**: 위 RPC 함수는 `SECURITY DEFINER` + RLS 조합으로만 실행. 프론트엔드 직접 UPDATE 절대 금지.

## 4. 어드민(관리자) 컨트롤 패널 연동

### 4.1 아키텍처 개요

```
관리자 UI (admin/page.tsx)
    │
    ├── platform_settings 테이블  ──→  런타임 정책 설정 (최대 견적 수, Credits 비용 등)
    │
    ├── cash_ledger 뷰  ──→  모든 Credits 이동 이력 감사
    │
    ├── referral_rewards 테이블  ──→  추천 보상 현황 및 이상 패턴 탐지
    │
    └── admin_manage_cash() RPC  ──→  Credits 수동 조정 (어뷰징 계정 제재)
```

### 4.2 관리자 조정 가능 설정 (platform_settings)

| 설정 키                    | 기본값    | 설명                                                                                                     |
| -------------------------- | --------- | -------------------------------------------------------------------------------------------------------- |
| `max_quotes_per_request`   | 5         | 요청서당 최대 견적 발송 수                                                                               |
| `referral_enabled`         | true      | 추천 프로그램 ON/OFF                                                                                     |
| `referral_bonus_credits`   | 150       | 추천 성공 시 지급 Bonus Credits                                                                          |
| `referral_coupon_amount`   | 200       | 추천 쿠폰 금액                                                                                           |
| `coupon_expiry_days`       | 365 (1년) | 쿠폰 만료 기간 (`coupons.expires_at` 기반, 만료 집행 cron 필요)                                          |
| `bonus_credit_expiry_days` | 365       | **현재 미사용** — Bonus Credits 만료 메커니즘 미구현 상태. 향후 유저 규모 확대 시 별도 스프린트에서 검토 |

> **코딩 규칙**: 위 값들을 코드에 직접 하드코딩 금지. 반드시 `platform_settings` 테이블에서 조회.

### 4.3 관리자 어뷰징 탐지 및 대응 흐름

```
1. 어뷰징 탐지 (admin/page.tsx 대시보드)
   - 비정상 Credits 변동 패턴 확인 (cash_ledger)
   - 추천 중복 보상 시도 확인 (referral_rewards)
   - 단기 대량 견적 발송 확인 (match_quotes)

2. 제재 조치
   A. Credits 수동 차감: admin_manage_cash(user_id, -amount, reason)
   B. 계정 정지: users.status = 'SUSPENDED' (DB 직접 또는 관리자 RPC)
   C. 추천 보상 취소: referral_rewards.status = 'REVOKED'

3. 감사 로그
   - 모든 관리자 조치는 cash_ledger (admin tx_type) 또는 별도 audit_log 테이블에 기록
   - 조치 사유 (reason) 필드 필수 입력
```

### 4.4 에러 디버깅 파이프라인 (어뷰징 관련 버그 발생 시)

```
1. DB/RLS → 2. 백엔드 RPC 쿼리 → 3. 캐시 → 4. 프론트엔드
```

- Credits 차감 이상: `cash_ledger` 원장 먼저 확인 → RPC 로직 → 프론트 상태
- 견적 수 초과 오류: `match_quotes` COUNT 확인 → `send_quote_and_deduct_cash()` 내 체크 로직
- 추천 보상 미지급: `referral_rewards` 레코드 확인 → `process_referral_reward()` RPC 실행 로그

## 5. 상태 전환 규칙 (State Transition Spec)

**절대 규칙**:

- 상태 전환은 반드시 DB RPC 함수 또는 백엔드 API를 통해서만 수행
- 프론트엔드 직접 `UPDATE match_requests SET status = 'MATCHED'` 금지
- 상태 전환 시 관련된 모든 테이블을 원자적으로 업데이트 (트랜잭션 보장)

### 5.1 고객 화면 (match_requests 상태 → 탭 분류)

| 상태 / 조건                                                         | 올바른 탭      |
| ------------------------------------------------------------------- | -------------- |
| OPEN (견적 0개 / 대기 중)                                           | 진행 중인 견적 |
| OPEN + 견적 있음                                                    | 진행 중인 견적 |
| 견적 최대 개수 도달 (현재 5개 설정)                                 | 진행 중인 견적 |
| MATCHED (확정 완료)                                                 | 진행 중인 견적 |
| 리뷰 작성 완료                                                      | 마감된 견적    |
| MATCHED 확정 후 30일 경과 (리뷰 미작성 시 자동 마감)                | 마감된 견적    |
| CANCELED (삭제 활성)                                                | 마감된 견적    |
| 48시간 만료 (견적 요청 후 48시간 내 미매칭/미결정 시 자동 타임아웃) | 마감된 견적    |

### 5.2 고수 화면 (match_quotes 상태 → 탭 분류)

| 상태 / 조건                                  | 올바른 탭 (위치)            | 동작 기준                                                                                    |
| -------------------------------------------- | --------------------------- | -------------------------------------------------------------------------------------------- |
| 요청서 발생 (내 견적 미발송)                 | 새로운 요청                 | 견적을 발송하는 순간 이 리스트에서 사라짐                                                    |
| 견적 발송 완료 (대기 중)                     | 보낸 견적 > 진행중          | 발송 즉시 대기 상태로 '진행중' 탭에 들어감                                                   |
| 내가 낙찰됨 (ACCEPTED)                       | 보낸 견적 > 진행중          | 매칭 성공 시 실제 작업 수행을 위해 '진행중'에 유지됨                                         |
| 고객의 리뷰 작성 완료                        | 보낸 견적 > 보관함          | 고객이 최종 리뷰를 남기면 서비스가 완료된 것으로 간주하여 보관함으로 이동                    |
| 낙찰(ACCEPTED) 후 30일 경과                  | 보낸 견적 > 보관함          | 리뷰 미작성 건의 무한 적재를 막기 위해 확정 후 30일 경과 시 자동 이동                        |
| 다른 고수 낙찰 (나는 탈락)                   | 보낸 견적 > 보관함          | 다른 고수가 낙찰(MATCHED)되고 나는 탈락(IACCEPTED)한 경우 보관함으로 이동                    |
| 고객 무응답 48시간 만료                      | 보낸 견적 > 보관함          | 견적 발송 후 고객이 48시간 내에 아무도 낙찰하지 않아 만료된 경우 보관함으로 이동             |
| 고객이 요청서 취소 / 낙찰 후 취소 (CANCELED) | 보낸 견적 > 보관함          | 낙찰 전/후 관계없이 요청이 취소된 경우 보관함으로 이동                                       |
| 보관함 이동 후 7일 경과                      | 숨김 처리 (화면 노출 안 됨) | 기준 시각(updated_at) 7일 경과 시 DB 쿼리 레벨 필터링 + 프론트엔드 이중 방어로 화면에서 숨김 |

## 6. 추천인 시스템 (Referral System)

### 6.1 4-Way 보상 구조

| 추천 방향           | 추천인 보상   | 피추천인 보상 |
| ------------------- | ------------- | ------------- |
| Pro → Pro           | Bonus Credits | Bonus Credits |
| Customer → Pro      | 쿠폰          | Credits       |
| Customer → Customer | 쿠폰          | 쿠폰          |
| Pro → Customer      | Credits       | 쿠폰          |

### 6.2 보상 지급 조건

- **지급 시점**: 피추천인의 **첫 견적 발송**(고수) 또는 **첫 견적 요청**(고객) 완료 시
- **가입만으로는 지급 불가** — 실제 서비스 이용 행위가 반드시 선행되어야 함

### 6.3 쿠폰 Closed-loop 구조

```
고객이 고수에게 쿠폰 코드 제시
    → 고수가 앱에서 쿠폰 코드 등록
    → 시스템이 Bonus Credits으로 전환
    → 외부 현금 유출 0
```

### 6.4 어뷰징 방어

| 방어 항목            | 구현 방법                                                     |
| -------------------- | ------------------------------------------------------------- |
| 자기 추천 불가       | DB: `referred_by != referrer_id` CHECK 제약                   |
| 중복 보상 불가       | DB: `referral_rewards` UNIQUE (referrer_id, referred_user_id) |
| 가입만으로 보상 불가 | 트리거: 첫 견적 발송/요청 완료 후에만 지급                    |
| 계정 중복 생성 방지  | 전화번호 인증 필수 (SMS OTP 검증)                             |

### 6.5 관리자 조정 가능 설정 (platform_settings)

| 설정 키                    | 기본값    | 설명                                                                                                     |
| -------------------------- | --------- | -------------------------------------------------------------------------------------------------------- |
| `referral_enabled`         | true      | 추천 프로그램 ON/OFF                                                                                     |
| `referral_bonus_credits`   | 150       | 추천 성공 시 지급 Bonus Credits                                                                          |
| `referral_coupon_amount`   | 200       | 추천 쿠폰 금액                                                                                           |
| `coupon_expiry_days`       | 365 (1년) | 쿠폰 만료 기간 (`coupons.expires_at` 기반, 만료 집행 cron 필요)                                          |
| `bonus_credit_expiry_days` | 365       | **현재 미사용** — Bonus Credits 만료 메커니즘 미구현 상태. 향후 유저 규모 확대 시 별도 스프린트에서 검토 |

> **코딩 금지**: 위 값들을 코드에 하드코딩 금지 — 반드시 `platform_settings` 테이블에서 런타임 조회

## 7. MVP 기획 핵심 정책 통합 (비즈니스 및 운영)

### 7.1 고수 서비스 등록 제한

무분별한 심부름센터화를 막기 위해, 고수가 프로필에 등록할 수 있는 '제공 서비스(3뎁스 카테고리)'는 **최대 5개**로 엄격히 제한합니다.

### 7.2 100% 직거래 원칙

- 플랫폼 내 에스크로 결제를 전면 배제하고, 서비스 대금은 현장 직거래로 유도합니다.
- 플랫폼은 오직 '견적 발송용 캐시(Credits)'만 관리합니다.

### 7.3 리뷰 및 소통 권한

- **리뷰 권한**: `MATCHED`(매칭 확정) 상태로 전환된 단 한 명의 진짜 고객만 1방향 리뷰를 남길 수 있습니다.
- **필터링 배제**: 프로필 및 채팅창에서 전화번호나 외부 링크 공유를 차단하는 필터링 로직은 전면 배제합니다.
